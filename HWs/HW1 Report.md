## Условие - Making The Grade
Очень большой и густонаселенный субъект РФ хотел бы внедрить новую систему для проведения стандартизированного тестирования во всех государственных школах с 2-го по 11-й класс.

Пользователи: более 40 000 учащихся, 2000 учителей, 50 администраторов.

Требования:
- Учащиеся смогут пользоваться приложением только в центрах тестирования по всему субъекту, большинство из которых расположены в школах, но не все.
- Учащиеся должны иметь возможность пройти тест, а результаты в конечном итоге должны быть собраны в одном месте, где будут представлены все результаты тестирования по всему субъекту (по школам, учителям и учащимся).
- Тесты будут включать в себя вопросы с несколькими вариантами ответа, вопросы с краткими ответами и эссе.
- В системе должна быть предусмотрена система отчетности, позволяющая отслеживать, кто из учащихся прошел тестирование и какие баллы получил.
- Вопросы с краткими ответами и эссе будут оцениваться вручную учителями, которые затем будут вносить оценки в систему.

Дополнительный контекст:
- для внесения изменений в систему хранения оценок учащихся требуется одобрение трех различных государственных ведомств, что обеспечивает безопасность данных.
- Субъект РФ не владеет собственным хостинговым центром, а передает его на аутсорсинг сторонней организации.
- Каждый финансовый год проект должен подтверждать свой бюджет.
- Вы находитесь в России, 2026 год

## Сбор и анализ требований
### Отсев
Реализуемые требования:

|-------|--------------------------------------------------------------|

| R-3    | Три типа вопросов (множественный выбор, краткий ответ, эссе) |

| R-7    | Хостинг на аутсорсе (нет собственного ЦОД)                   |
| R-8    | Ежегодное подтверждение бюджета                              |

Вне области внедрения (out-of-scope):
| ID    | Что отсеяно                                                  | Причина |
|-------|--------------------------------------------------------------|---------|
| OOS-1 | Прохождение тестов вне центров тестирования (из дома, с личных устройств) | Прямо противоречит требованию «только в центрах»; контролируемая среда обязательна |
| OOS-2 | Мобильное приложение для учащихся                            | Тестирование на десктопах в центрах; мобильный клиент не нужен и расширяет attack surface |
| OOS-3 | Автоматическая проверка эссе и кратких ответов (в т.ч. ML/LLM-оценивание) | По условию эти работы проверяются вручную учителями |
| OOS-4 | Прокторинг с видеонаблюдением / распознаванием лиц           | Не заявлено; усиливает требования к ПДн несовершеннолетних, выносится за рамки первой версии |
| OOS-5 | Интеграция с внешними информационными системами образования | В условии не заявлена; рассматривается как отдельный этап |
| OOS-6 | Самостоятельное создание тестов учителями                   | Банк вопросов и сборка тестов - централизованная функция администрирования |

### Функциональные требования
Акторы:

Основные сущности:
| Объект | Описание                                                                    |
|--------|-----------------------------------------------------------------------------|
| Вопрос  | Вопрос из банка вопросов с указанием типа (множественный выбор, краткий ответ, эссе), текста и максимального первичного балла |
| Тест    | Сформированный тест, включающий набор вопросов, ограничение по времени и привязку к шкале перевода баллов |

| Центр тестирования | Оборудованная площадка, где учащиеся проходят тесты | 
| Критерий оценивания эссе | Критерий с максимальным баллом по данному критерию |

### Нефункциональные требования
#### Производительность
| Метрика | Значение | Обоснование |
|---------|----------|-------------|
| Время отклика страницы (P95) | ≤ 2 секунды | Для комфортной работы учеников и учителей |
| Время сохранения ответа (автосейв) на локальный сервер (P95) | ≤ 200 мс | По локальной сети задержка минимальна, не должен мешать процессу |
| Время сохранения ответа (автосейв) в ЦОД (P95) | ≤ 500 мс | При наличии связи, фоновая синхронизация |
| Время автопроверки MC-вопросов (ЦОД) | ≤ 5 минут после синхронизации | |
| Время формирования отчёта по школе | ≤ 1 минуты | Допустимо для административных задач |
| Время формирования сводного отчёта по субъекту | ≤ 20 минут | Тяжёлый аналитический запрос, допустима задержка |
| Время синхронизации дневного объёма одного центра | ≤ 30 минут | При восстановлении связи |
#### Масштабируемость
| Аспект | Требование |
|--------|-----------|
| Горизонтальное масштабирование ЦОД | Система должна масштабироваться при росте числа школ и учеников (до 100 000 учащихся) без изменения архитектуры |
| Пиковая нагрузка на ЦОД | Одновременная синхронизация со всех 200 локальных серверов |
| Пиковая нагрузка на локальный сервер | До 200 учеников одновременно в одном центре |
| Рост данных | Хранение данных минимум 5 лет с ростом ~20% в год |
| Масштабирование по центрам | Добавление нового центра тестирования - установка локального сервера и регистрация в ЦОД, без доработки системы |

##### Пиковый сценарий: массовая синхронизация после восстановления связи

В штатном режиме данные за день накапливают ~40 центров. Худший случай - редкое событие: региональный сбой связи в дни волны, когда офлайн-бэклог накапливается у большинства центров и выгружается одновременно при восстановлении. Берём консервативную верхнюю границу - одновременная синхронизация всех 200 центров.

#### Ключевые выводы по нагрузке

1. Система средненагруженная с ярко выраженной сезонностью - пик в дни тестирования, минимум между волнами
2. Главный вызов - не абсолютная нагрузка, а надёжность: ни один ответ не должен быть потерян, тестирование нельзя сорвать
3. Двухуровневая архитектура (лок.сервер + ЦОД) решает проблему нестабильного интернета - ученики работают только с локальной сетью
4. Автомасштабирование облачных ресурсов ЦОД позволяет экономить бюджет между волнами
5. Пик нагрузки - массовая синхронизация 200 серверов после восстановления связи, ~1 430 RPS, покрывается 2 серверами приложений
6. Локальные серверы имеют минимальные требования к оборудованию - существующие ПК в школах подходят
